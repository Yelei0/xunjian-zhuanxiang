# 危化品运输行业 PRD 生成

## 目标

生成一份可人工维护、可被页面需求说明组件解析的 Markdown PRD。PRD 既是产品文档，也是 `annotation-panel` 的数据源。

核心要求：

- 仍然输出 Markdown，方便用户直接修改。
- 页面需求说明不得写死在 Vue 页面里，必须来自 PRD。
- 每个页面的需求编号独立从 `[1]` 开始。
- 每个可标注功能点必须能和页面 DOM 的 `data-req-id` 对应。
- 页面级说明、状态机、特殊情况等不绑定具体角标，但必须能进入需求说明面板。
- Markdown 表格必须写成标准表格，方便前端渲染为 HTML table。

## 前置步骤

1. 确认业务模块范围和涉及页面。
2. 读取每个页面的 `index.vue`、`spec.md`、`design.md`。
3. 读取 `skills/industry-data-rules.md`，确保模拟数据、字段、行业场景合规。
4. 识别每个页面的 `data-req-page` 和 `data-req-id`。PRD 编号必须与页面标注一致。
5. 移动端 PhoneFrame 内部多个子页面必须拆成多个 PRD 页面章节。

## PRD 顶层结构

```markdown
# [业务模块名称] PRD

> 本文档作为原型页面需求说明的数据源。每个页面的需求编号独立从 1 开始。

## 一、[页面名称]

**页面**：inspection-record
**类型**：后台列表
**菜单路径**：园区巡检 / 巡检记录
**页面说明**：用于查询、导出、查看和维护园区巡检记录。
**需求数**：7

### 页面功能描述

### 页面状态机

### 页面特殊情况

### [1] 筛选区 {type=inline, target=filter-section}

### [2] 巡检详情 {type=modal, modalId=detailModal, trigger=req:1.action:view-detail}
```

## 页面元信息规则

每个页面章节必须包含以下元信息，且字段名保持一致：

| 字段 | 必填 | 说明 |
|------|------|------|
| 页面 | 是 | 对应页面根节点 `data-req-page`，例如 `inspection-record` |
| 类型 | 是 | 数据看板 / 后台列表 / 表单编辑 / 移动端 |
| 菜单路径 | 否 | 后台页面写菜单路径，移动端写入口路径 |
| 页面说明 | 是 | 页面整体业务职责，不写 UI 布局 |
| 需求数 | 是 | 当前页面功能点数量，不包含页面级说明 |

页面标题、页面元信息中的 `页面`、页面 DOM 的 `data-req-page` 必须一一对应。

## 页面级说明

页面级说明用于承载不适合绑定到某一个角标的内容。每个页面章节建议包含以下三块：

### 页面功能描述

说明页面业务目标、业务边界、主要用户和核心处理对象。不要描述颜色、布局、组件实现。

### 页面状态机

当页面涉及任务、审批、整改、提交、异常处理等状态流转时必须填写。

| 状态 | 业务含义 | 进入条件 | 可执行操作 | 后续状态 |
|------|----------|----------|------------|----------|
| 未提交 | 已创建但尚未提交结果 | 暂存后生成 | 继续填写、提交 | 已提交 / 已完成 |

无状态流转时写“无独立状态机”。

### 页面特殊情况

说明空数据、加载失败、提交失败、权限不足、异常数据、边界条件等。

| 场景 | 触发条件 | 处理规则 |
|------|----------|----------|
| 空数据 | 当前筛选条件无结果 | 展示空状态并允许重置筛选 |

## 功能点编号和 DOM 映射

### 编号规则

- 每个页面独立从 `[1]` 开始编号。
- 同一页面内编号唯一。
- 需求标题格式必须是 `### [N] 需求名称 {元信息}`。
- 页面 DOM 使用 `data-req-id="N"` 对应 PRD 编号。
- 同一个需求可绑定多个 DOM 元素，例如查询和重置按钮共同属于 `[2]`。
- 页面有 `data-req-id` 但 PRD 没有对应编号时，必须回补 PRD。
- PRD 有编号但页面没有对应 `data-req-id` 时，必须在页面或设计说明中标记为未绑定。

### 元信息规则

需求标题后可追加 `{key=value}` 元信息：

| key | 必填 | 示例 | 说明 |
|-----|------|------|------|
| type | 否 | `inline` / `modal` | 默认 `inline` |
| target | 否 | `filter-section` | 功能区域语义标识，辅助人工维护 |
| modalId | 弹窗必填 | `detailModal` | 对应页面监听的弹窗标识 |
| trigger | 弹窗建议填 | `req:4.action:view` | 说明弹窗由哪个功能点或操作触发 |
| scenarios | 否 | `[预览异常:scan-error, 预览成功:scan-success]` | 面板中的场景触发按钮 |

示例：

```markdown
### [5] 巡检详情 {type=modal, modalId=detailModal, trigger=req:4.action:view-detail}
```

## 功能点内容格式

每个功能点至少包含“功能说明”。涉及字段、操作、校验、状态变化时必须用表格表达。

推荐结构：

```markdown
### [N] 功能点名称 {type=inline, target=xxx}

#### 功能说明
说明该功能点的业务职责、前置条件和结果。

#### 字段规则
| 字段 | 类型 | 是否必填 | 默认值 | 长度/范围 | 校验规则 | 备注 |
|------|------|----------|--------|-----------|----------|------|

#### 操作规则
| 操作 | 触发位置 | 前置条件 | 成功结果 | 失败/异常处理 |
|------|----------|----------|----------|----------------|

#### 数据规则
| 数据项 | 规则 |
|--------|------|

#### 状态变化
| 当前状态 | 操作 | 条件 | 下一状态 |
|----------|------|------|----------|
```

不适用的表格可以省略，但表单字段、筛选字段、列表字段、弹窗字段必须使用表格。

## 弹窗和触发开关设计

弹窗、抽屉、确认框、ActionSheet 等需要被标注面板触发的内容，必须作为独立需求点。

弹窗类需求必须写：

```markdown
### [N] 删除确认 {type=modal, modalId=deleteModal, trigger=req:4.action:delete}
```

弹窗内容必须包含：

| 小节 | 要求 |
|------|------|
| 触发规则 | 谁触发、前置条件、打开哪类弹窗 |
| 展示内容 | 弹窗内字段、提示文案、只读/可编辑内容 |
| 操作规则 | 确认、取消、关闭、异常处理 |
| 状态变化 | 如果会改变业务状态，必须说明 |

如果一个按钮只用于预览某个业务状态，不打开弹窗，可使用 `scenarios`：

```markdown
### [2] 点位核对与打卡 {type=inline, target=scan-check, scenarios=[预览异常:scan-error, 恢复成功:scan-success]}
```

场景名称用于面板按钮展示，场景值用于页面监听 `annotation:apply-scenario`。

## 字段规则要求

生成 PRD 时不得只写“填写基础信息”。涉及表单、筛选、弹窗输入、移动端填写项时，必须输出字段表。

字段表至少包含：

| 字段 | 类型 | 是否必填 | 默认值 | 长度/范围 | 校验规则 | 备注 |
|------|------|----------|--------|-----------|----------|------|

填写规则：

- “是否必填”只写“是/否/条件必填”。
- “默认值”没有默认值时写“无”，不要留空。
- “长度/范围”写最大长度、最小值、枚举范围、数量上限等。
- “校验规则”写业务校验，例如手机号格式、结束时间不得早于开始时间、至少上传 1 张照片。
- 原型阶段不写接口字段名、数据库字段名、API 参数。

## 列表和看板规则

列表页必须补充：

- 筛选字段表。
- 列表字段表。
- 行操作和批量操作表。
- 分页、排序、导出范围。
- 空数据、加载、异常状态。

看板页必须补充：

- 指标口径表。
- 图表口径表。
- 时间范围和刷新规则。
- 告警阈值和状态说明。
- 下钻或联动规则。

## 移动端规则

移动端内部子页面按独立页面处理。即使多个手机页面写在同一个 `index.vue` 中，也必须拆成多个 PRD 页面章节。

推荐页面标识：

| PRD章节 | DOM 页面标识 |
|--------|-------------|
| 移动端巡检首页 | `mobile-inspection` |
| 移动端开始巡检 | `mobile-inspection-start` |
| 移动端巡检任务 | `mobile-inspection-task` |
| 移动端巡检记录 | `mobile-inspection-records` |
| 移动端记录详情 | `mobile-inspection-detail` |
| 移动端点位地图 | `mobile-inspection-map` |

移动端 PRD 必须说明：

- 页面入口和返回目标。
- 步骤流转，例如下一步、暂存、提交后跳转。
- 拍照、扫码、签名等原型模拟行为。
- 弱网、未完成、异常点位、继续填写等特殊情况。

## 权限规划

PRD 末尾保留统一权限规划，不在页面中定义具体角色。

| 权限点 | 覆盖功能 |
|--------|----------|
| 查看 | 查看列表、详情、看板、地图 |
| 编辑 | 新增、编辑、提交、暂存、整改 |
| 删除 | 删除记录、删除配置 |
| 导出 | 导出当前筛选范围的数据 |

## 全局交互

PRD 末尾保留全局交互，写跨页面通用规则：

1. 所有删除操作必须二次确认。
2. 所有表单保存前必须校验必填项。
3. 数据刷新、提交、暂存、导出均为前端模拟行为，不对接真实后端。
4. 弹窗取消时不改变原数据。
5. 加载、空数据、异常状态必须在页面中体现。

## 输出

默认输出到项目根目录 `PRD.md`。如果用户指定模块文件，可输出到 `prd/<模块名称>.md`。

## 自检清单

生成完成后必须检查：

- 页面元信息是否齐全。
- 每个页面编号是否从 `[1]` 开始且无重复。
- 每个 `data-req-id` 是否有对应 PRD 编号。
- 弹窗需求是否包含 `type=modal` 和 `modalId`。
- 触发型弹窗是否说明 `trigger`。
- 表单、筛选、弹窗字段是否有完整字段规则表。
- Markdown 表格是否标准，列数是否一致。
- 是否误写 UI 实现、CSS、Vue 组件、接口字段、数据库字段。
- 是否符合 `skills/industry-data-rules.md`。
## PRD 可编辑块协议

锚点：`prd-editable-blocks`、`stable-heading-and-id`。

PRD 不只是一次性文档，也是需求说明面板的可维护数据源。生成 `PRD.md` 时必须保证页面级说明和功能点说明可以被面板稳定定位、编辑、写回。

规则：
- 默认输出到项目根目录 `PRD.md`，除非用户明确指定其它 PRD 文件。
- 页面级说明块标题必须稳定，例如 `### 页面功能描述`、`### 页面状态机`、`### 页面特殊情况`。
- 功能点标题必须稳定使用 `### [N] 功能点名称 {元信息}`，编号 `[N]` 不得因为后续补充说明随意重排。
- `stable-heading-and-id`：标题层级、页面标识、需求编号是写回定位依据，不要生成同一页面内重复 `[N]`。
- 正文可以被用户在需求说明面板内微调，因此不要把多个页面或多个功能点的内容混在同一个 Markdown 块里。
- 表格必须使用标准 Markdown 表格，便于面板渲染和编辑后继续解析。
- 页面功能描述、状态机、特殊情况等不绑定具体 `data-req-id`，但必须独立成块，便于三击标题编辑。
- 弹窗和场景类需求必须保留 `type=modal`、`modalId`、`trigger` 或 `scenarios` 元信息，便于面板触发预览和后续截图/导出。
